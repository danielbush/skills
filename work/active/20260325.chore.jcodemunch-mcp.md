# chore: jcodemunch skill

Build a skill that gives agents code search and exploration via jcodemunch-mcp, using `bunx mcporter` instead of registering jcodemunch as an MCP server.

## Key decision: skill over MCP

**Problem**: jcodemunch registered as MCP in `.mcp.json` wasn't being used proactively — the human always had to explicitly ask the agent to use it.

**Root cause**: MCP tool schemas (all 25 tools, full parameter definitions) load into context on *every API request* for the entire session. Despite this permanent context cost, the agent defaults to Read/Grep/Glob because those are its built-in "instincts." Having tools available doesn't mean the agent reaches for them.

**Decision**: replace MCP registration with a skill. Benefits:
- **Context**: only the skill description (~100 words) sits in context permanently. The full SKILL.md loads only when triggered. MCP loads all 25 tool schemas on every request.
- **Triggering**: the skill description explicitly tells the agent *when* to use code intelligence. MCP tools just sit there waiting to be noticed.
- **Portability**: the skill installs into any project. MCP requires `.mcp.json` per-project.

**Tradeoff**: skill uses `bunx mcporter call` (Bash) instead of native tool calls. Slightly more indirect, but the trigger reliability matters more than call ergonomics.

## Trigger eval

**Eval set**: `skills/jcodemunch/evals/trigger_eval.json` — 10 should-trigger (code exploration, symbol lookup, dependency questions) + 10 should-not-trigger (writing code, debugging, config, CI, git).

**Results**: 10/20 passed — **0% trigger rate on ALL should-trigger queries** (0/30 attempts). All should-not-trigger queries correctly didn't fire (10/10 passed).

The skill never triggered once. Claude handles code exploration queries ("where is X", "show me dependencies") with built-in tools (Grep/Read/Glob) and never considers consulting a skill. Per skill-creator docs: "Claude only consults skills for tasks it can't easily handle on its own." Code search is something Claude thinks it can do natively.

**Implication**: skill triggering is designed for capabilities Claude doesn't have, not for "use this better tool instead of your built-in ones." The description doesn't matter — no description will make Claude reach for a skill when it already has Read/Grep.

## Possible directions

### Direction 1: CLAUDE.md instructions + MCP

Keep jcodemunch as an MCP server (`.mcp.json`) but add instructions to the project's CLAUDE.md telling the agent to use it proactively. The skill becomes reference-only (or is dropped).

```markdown
## Code exploration

Before reading files directly, use jcodemunch MCP tools to orient:
- `search_symbols` to find what you're looking for
- `get_file_outline` to understand a file's structure before reading it
- `get_dependency_graph` / `find_references` to understand relationships
- `get_blast_radius` before making changes

These give you symbol-aware intelligence that Grep/Read can't — use them first.
```

**Pro**: CLAUDE.md is always loaded, so the instruction is always present. MCP tools are native calls — no Bash detour.
**Con**: MCP schemas still eat context on every request. CLAUDE.md instructions are softer than tool availability — the agent may still default to Read/Grep. Untested whether this is sufficient.

### Direction 2: Hybrid — CLAUDE.md push + skill for reference

Keep the skill installed for its reference material (tool surface docs, workflow patterns, mcporter syntax). Add CLAUDE.md instructions that push the agent toward jcodemunch. Drop the MCP registration — the skill provides access via mcporter when the agent does reach for it.

```markdown
## Code exploration

Use `/jcodemunch` or consult the jcodemunch skill for symbol-aware code search,
dependency analysis, and codebase orientation. Prefer this over raw Grep/Read
when exploring code structure, relationships, or impact of changes.
```

**Pro**: no MCP context cost. Skill reference is available when needed. CLAUDE.md provides the nudge.
**Con**: two mechanisms (CLAUDE.md + skill) for one thing. The nudge may still not be enough — if the eval showed 0% auto-trigger, a CLAUDE.md mention may also be ignored.

### Direction 3: Explicit slash command (`/jcodemunch`)

Make the skill a slash command the human invokes directly: `/jcodemunch where is parseToken defined`. Add a "When invoked" section to SKILL.md that reads the ARGUMENTS, determines which jcodemunch tools to call, runs them via mcporter, and presents results. No auto-trigger — the human controls when code intelligence is used.

**Pro**: reliable — fires exactly when asked. No MCP context cost. The skill does the tool selection (human doesn't need to know which of the 25 tools to use). Matches how the human already works ("I need to look something up").
**Con**: fully manual — the agent never proactively uses code intelligence. Misses cases where the agent *should* look things up but doesn't think to. Adds friction compared to the agent just knowing.

### Which to try?

Direction 3 is the most reliable but least ambitious. Direction 1 is the most natural but unproven (CLAUDE.md may not be strong enough either). Direction 2 hedges but may end up with neither mechanism working well. Could test Direction 1 by adding a CLAUDE.md instruction and re-running the eval with MCP active.
