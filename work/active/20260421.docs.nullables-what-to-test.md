# docs: Refine "What To Test" in nullable-architecture

Rewrote the section in `skills/nullable-architecture/SKILL.md` under "What To Test".

## Changes

- **2026-04-21** — Replaced the "avoid redundant tests on intermediate code" paragraph with:
  - Explicit scope: this section is about **unit tests**.
  - Interaction criterion: avoid tests that exercise interactions between system components to produce a behaviour. They lock in implementation details — refactor the interaction, rewrite the test. Instead test the component that performs the interaction.
  - Diagnostic question: "did the test have to perform several actions on several subcomponents to achieve a single outward behaviour?" If yes, it's probably encoding internals.
  - Barbell framing: unit tests should be either (a) high-level sociable (nullables + DI make this practical) or (b) low-level narrow (single-responsibility unit). Avoid the middle.
  - Smell: mirroring the same behaviour at multiple levels. With nullables + DI, the top-level test already exercises lower layers — mirroring is redundant.
