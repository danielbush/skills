# discussion: Concrete nullables skills

Supersedes NULLABLES_SKILLS__WORK. The previous approach tried to define the world upfront with vocabulary and classification taxonomies. New direction: small skills that each do one specific thing, individually activatable but also referenceable from CLAUDE.md / AGENTS.md so a project can opt in.

## Skill ideas

### 1. Replace embedded deps with injected ones

Extract anything that talks to the outside world and inject it instead. "Outside world" includes obvious things (filesystem, network, database) but also in-memory environment stuff like the browser — user interactions, DOM, timers, etc.

### 2. State-based testing (trackX)

Test by inspecting state after the fact, not by asserting calls were made. Uses `trackX` pattern to make internal state observable without mocks. **Never use interaction-based tests. Never use mocks.** State-based testing and behaviour simulation are the two testing tools — mocks and spy-style assertions are off the table entirely.

### 3. Behaviour simulation

Simulate events from the outside world — user clicks, incoming messages, timer fires, etc. The test drives behaviour by producing the stimuli, not by mocking the response.

## Assumed conventions (cut across all skills)

These aren't skills themselves but conventions that every skill above assumes:

- **`.create` and `.createNull` factories** — production instances via `.create`, test instances via `.createNull` that wire up nullable deps
- **`new` for thing under test, `.createNull` for dependencies** — when testing Foo, use `new Foo(Bar.createNull())`, not `Foo.createNull()`. `.createNull` is for when Foo is someone else's dependency.

## Current state

Replaced the two previous skills (nullables-refactor, nullables-test) with a single lighter skill: `skills/nullables-testing-style/SKILL.md`. This skill covers the full picture — infrastructure wrappers, construction rules, output tracking, behaviour simulation, state-based assertions — in one practical document with concrete examples rather than taxonomy.

**Next step:** Try it on real code (the other project with mock-based tests) and give feedback on what works, what's missing, and what's awkward.
